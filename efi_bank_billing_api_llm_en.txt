## Overview

The **EFI Billing API** offers advanced resources to issue different types of charges:
- Bank slip (boleto banc√°rio)
- Credit card
- Installment plan (carn√™)
- **Payment links** (focus of this documentation)
- Subscriptions (recurring charges)
- Marketplace (payment splitting)
- **PIX** (integrated in links)

### Payment Link Features

The **Payment Link** resource allows:
- Generate a link to EFI's hosted payment page
- Choose allowed payment methods (bank slip, card, PIX)
- No need to implement your own checkout
- Provides security to the payer with EFI's domain

### Advantages over Transparent Checkout

- **Simplicity**: No need to develop your own interface
- **Security**: Trusted EFI interface
- **Flexibility**: Choose payment methods
- **Maintenance**: Updates managed by EFI

To integrate with the API, you need an **EFI Digital Account** and create an application.

**Process**:
1. Access your EF√≠ account
2. Click on "API" in the side menu
3. Click on "Create application"

- `Client_Id` and `Client_Secret` for **Production**

## Payment Link - Endpoints

### Approaches: One Step vs Two Steps

The API offers two approaches to create payment links:

#### **One Step** (Recommended for simplicity)
- Creates transaction and link in a single request
- Returns `payment_url` directly
- Initial status: `link`

#### **Two Steps** (More control)
1. Create transaction (`POST /v1/charge`) ‚Üí receives `charge_id`, status: `new`
2. Associate with link (`POST /v1/charge/:id/link`) ‚Üí receives `payment_url`, status: `link`

## Notification System (Webhooks)

### How It Works

EFI sends **automatic notifications** via HTTP POST whenever a transaction status changes.

**2-Step Process**:
1. **EFI ‚Üí Your URL**: POST with `notification` token
2. **You ‚Üí EFI**: GET `/v1/notification/:token` to get details

### Configure Notification URL

**During transaction creation**:
```json
{
  "items": [...],
  "metadata": {
    "notification_url": "https://yoursite.com/webhook/efi"
  }
}
```

**Update later**:
```
PUT /v1/charge/:id/metadata
```

### Receive Notification (Webhook)

**EFI's Request to your URL**:
```
POST https://yoursite.com/webhook/efi
Content-Type: application/x-www-form-urlencoded

notification=09027955-5e06-4ff0-a9c7-46b47b8f1b27
```

**Your endpoint must**:
1. Respond with HTTP status `200 OK`
2. Extract the token: `$_POST['notification']`
3. Query details: `GET /v1/notification/:token`

### Query Notification Details

**Endpoint**:
```
GET /v1/notification/:token
```

**Headers**:
```
Authorization: Bearer {access_token}
```

**URL Params**:
- `:token` ‚Üí notification token received in POST

**Example Response**:
```json
{
  "code": 200,
  "data": [
    {
      "id": 1,
      "created_at": "2022-02-20 09:12:23",
      "type": "charge",
      "custom_id": null,
      "identifiers": {
        "charge_id": 24342333
      },
      "status": {
        "current": "new",
        "previous": null
      }
    },
    {
      "id": 2,
      "created_at": "2022-02-20 09:12:23",
      "type": "charge",
      "identifiers": {
        "charge_id": 24342333
      },
```

### Notification Response Structure

| Field | Description |
|-------|-------------|
| `id` | Sequential order of events (1, 2, 3...) |
| `created_at` | Date/time of change |
| `type` | Charge type ("charge", "subscription", "carnet") |
| `custom_id` | Your custom identifier |
| `identifiers.charge_id` | Unique transaction ID |
| `status.current` | Current status |
| `status.previous` | Previous status |
| `value` | Confirmed value (appears when `paid`) |
| `received_by_bank_at` | Received date (when `paid`) |

### Notification Order

‚ö†Ô∏è **IMPORTANT**: Notifications follow chronological order of events.

**Example - Installment Plan**:
- Installment 1 paid ‚Üí notification with `id: 1`
- Installment 2 paid ‚Üí notification with `id: 2`
- Installment 3 paid ‚Üí notification with `id: 3`

To sync: traverse the array and process unsynced events (don't assume last is always new).

## Best Practices

### ‚úÖ Security

1. **Protect your credentials**:
   - Use environment variables (`.env`)
   - NEVER commit `Client_Secret` to git
   - Implement 2FA

2. **Validate Webhooks**:
   - Always query `GET /v1/notification/:token`
   - Don't trust only the received POST
   - Implement logging of all notifications

3. **HTTPS Mandatory**:
   - Use SSL in production
   - Keep certificates updated
   - Configure redirects correctly (301/302)

### ‚úÖ Error Handling

1. **Always return HTTP 200** in webhook (even with internal error)
2. Implement retry logic for EFI API calls
3. Use try-catch in all API interactions
4. Maintain detailed error logs

### ‚úÖ Performance

1. **Webhook must be fast**:
   - Receive POST ‚Üí validate ‚Üí respond 200 OK
   - Process business logic in background (jobs/queues)

2. **Token caching**:
   - Tokens last 600 seconds (10 min)
   - Reuse valid tokens
   - Implement auto refresh

3. **Rate Limiting**:
   - Endpoint `/charge/:id/metadata`: max 7,500 req/24h
   - Implement request control

### ‚úÖ Monitoring

1. **Check Queue Status**:
   - Access: https://sejaefi.com.br/confirmacoes
   - Monitor callback delays

2. **Notification History**:
   - EFI interface shows sent POSTs
   - Check failures (404, 500, etc.)

3. **Test with Postman**:
   - Official collection: https://documenter.getpostman.com/view/13574984/Uz5ArJQc
   - Configure environments (Production/Sandbox)

### ‚úÖ Development

1. **Use Sandbox environment** for testing
2. **WebhookInbox** to test callbacks locally
3. Implement **feature flags** to enable/disable integrations
4. Keep **internal documentation** updated

### ‚úÖ Transaction Management

1. **Always use `custom_id`**:
   - Facilitates tracking
   - Avoids duplications
   - Enables reconciliation

2. **Monitor final status**:
   - `paid` ‚Üí process order
   - `canceled` ‚Üí notify user
   - `refunded` ‚Üí reverse operations

3. **Transaction history**:
   - Use `POST /v1/charge/:id/history` for notes
   - Record important changes
   - Facilitates customer support
